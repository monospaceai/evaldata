| 25 | [x] | No cancellation/timeout enforcement: `CostBudget.max_seconds` exists on `EvalCase` but nothing enforces it. When wiring it, add a `cancel()` method to the `PlatformAdapter` Protocol (dbt-style) so an in-flight query can be aborted when the budget is exceeded. **Done.** Locked design (research in subagents; dbt + driver primitives): (1) **mechanism** — background-thread + `cancel()`, not driver-native statement timeouts. dbt cancels in-flight queries from a separate thread and sets no native timeout; DuckDB has no clean per-statement timeout, so thread+cancel is the only approach that generalizes. (2) **`cancel()` added to the Protocol**; DuckDB calls `connection.interrupt()` (no-op when idle, doesn't raise), Postgres calls `connection.cancel_safe()` wrapped in `contextlib.suppress(psycopg.Error)` (libpq cancel over a separate channel; thread-safe; best-effort). Both interrupt the blocked `execute`, which raises a `driver.Error` subclass (`InterruptException` / `QueryCanceled`) already caught by `execute` → surfaces as `ExecutionResult.error`. (3) **enforcement lives in a reusable `execute_within_budget(adapter, sql, max_seconds)` helper in `platforms/base.py`** (keeps the Protocol minimal and adapters dumb), called from `runner.assert_eval`: runs `execute` on a worker thread, waits `max_seconds`, on overrun calls `adapter.cancel()` and returns a synthesized `ExecutionResult.error` (`"exceeded cost budget: query did not complete within Ns"`) with `latency_seconds` = actual elapsed; the cancelled query is awaited so its connection is released before returning. The error flows unchanged through `ResultSetEquivalence` → terminal. Verified: shared conformance tests (within-budget, exceeds-budget-is-cancelled, idle-cancel-is-safe) across both adapters via `under_test` — DuckDB path run locally (recursive-CTE slow query, cancelled in <0.5s); Postgres path (`pg_sleep`) via CI/e2e (Docker unavailable locally → skips cleanly). `ruff` + `ty` clean on changed modules. |

| 27 | [x] | Follow-ups from #22 (the `SqlType`/`nullable` additions). **Done. Locked scope: build the missing `ExpectationSuite` scorer (so `not_null` etc. actually run), route `expected_type` through `SqlType` (b), and DROP the `Column.nullable` short-circuit (a) — but keep the `nullable` field.** Reframed by research during the task: (1) **The (a) short-circuit is dropped on principle, not deferred.** GE (`expect_column_values_to_not_be_null`) and dbt (`not_null`) both *always value-scan* and never trust a declared flag to skip the check — for a verification tool, using the declaration to skip verification is backwards. Independently, the actual-side signal doesn't even exist on our drivers: DuckDB's `cursor.description[6]` is empirically always `None` (even for `NOT NULL` base columns) and psycopg3's `Column.null_ok` is hardcoded `return None`. (2) **`Column.nullable` is intentionally KEPT** (no revert of #22): it's real schema metadata (cf. Arrow `Field.nullable`, Spark `StructField.nullable`), future connectors (BigQuery `mode`, Snowflake) populate it from result metadata, and declared-nullability / DDL `NOT NULL` testing — sourced from the catalog (`information_schema`/`pg_attribute`), the sound producer the cursor description lacks — is its proper future home. (3) **(b):** `ColumnTypeExpectation.expected_type` is now a `SqlType` (string shorthand keeps `expected_type="INTEGER"` authoring; `raw`'s `min_length=1` preserves non-empty), canonicalised at ingestion by a new `EvalCase._canonicalize_expectation_types` validator — no longer cosmetic: it's consumed by the new scorer's `column_type` arm (compares via `SqlType.__eq__`). **Built:** `ExpectationSuiteScorer` (`scorers/expectation_suite.py`), evaluating all five `Expectation` kinds via one exhaustive `match`/`assert_never` (`row_count`, `column_presence`, `column_type`, `not_null` value-scan, `unique` with dbt NULL-as-equal semantics + unhashable-cell handling); absent target column → fail-with-explanation (errors-as-values, not raise); `result.error`/non-`ExpectationSuite` handled like `ResultSetEquivalence`; returns one aggregated `ScoreResult`. No runner/CLI change needed — scorers are passed explicitly (`assert_eval(..., scorers=[...])`); the CLI `run` just shells to pytest. Verified (DuckDB-only; no Docker): `pytest -m "not e2e"` 336 passed; `ruff` clean on `src`+`tests`; `ty` clean on changed source (test-only `SqlType`/`Schema` string-shorthand diagnostics match the existing `test_result_set_equivalence.py` baseline). **Deferred (real future work, now well-scoped):** declared-nullability / DDL `NOT NULL` verification sourced from the catalog — a *separate* assertion from the value-scan, not an optimization of it. See #28 for the structured-metadata follow-up. |

Locked design (research: GE `ExpectationValidationResult` per-kind `result` dict — `observed_value` for aggregate/type checks vs `unexpected_count` for value-scans; dbt run-results `failures` int + `status`): (1) **carrier = a typed `outcomes: list[ExpectationOutcome]` field on `ScoreResult`**, not `metadata`. The data *is* a depended-upon contract (a report consumer reads `.kind`/`.column`/`.count`), so burying it in `dict[str, Any]` would make the type lie; and a typed optional field is the exact mirror of the existing `diff` field — `ResultSetEquivalence` populates `diff`, `ExpectationSuiteScorer` populates `outcomes`, each leaving the other empty (no new pattern, completes the one already there).

## 0.9.0 - 2026-07-22
### Changed
* Replace optional success/error fields with separate success and failure results for scalar queries, SQL execution, solvers, Semantic Layer solvers, and case reports. Platform references now use per-platform configuration models, and benchmark totals are computed from case reports. These changes update public constructors and serialized shapes.

## 0.8.0 - 2026-07-20
### Added
* Pydantic Evals integration: use `SqlEquivalence` to score generated SQL inside an existing `Dataset` against a gold query or expected rows. Install `evaldata[pydantic-evals]`.
* Bounded per-platform connection pools: run warehouse-backed cases concurrently with configurable `PoolPolicy` limits, native query deadlines, cancellation, health checks, and safe cleanup.
### Fixed
* Use SQLGlot 30.12.0's corrected constant-minus-variable simplification and remove the local workaround.

## 0.7.0 - 2026-07-13
### Added
* BigQuery platform adapter: run SQL evals and data checks on BigQuery. Install `evaldata[bigquery]`.

## 0.6.0 - 2026-07-09
### Added
* Snowflake Cortex Analyst solver: evaluate a deployed Cortex Analyst through the usual scoring cascade. `CortexAnalystSolver` sends a natural-language question to the Cortex Analyst REST endpoint (reusing your Snowflake connection's session token) and returns the SQL it generates, which evaldata runs and scores against a gold query. Install the `cortex` extra.
* Export the extension surface from the top-level `evaldata` namespace: the `Scorer` and `Solver` protocols, `EquivalenceCheck`, `QueryRunner`, `ScoreContext`, the platform-ref builders (`duckdb_platform`, `sqlite_platform`, `postgres_platform`, `databricks_platform`, `snowflake_platform`), and the LLM value types a custom backend returns (`Usage`, `Completion`, `TextCompletion`, `StubLlm`).
* Concurrent solver calls in benchmark runs: `run_benchmark` and `run_metric_benchmark` take `max_concurrency` (and `bench`/`dbt-bench`/`sl-bench` take `--max-concurrency`) to fan model calls out on a thread pool. Execution and scoring stay serial in case order, and the default of 1 keeps today's behavior.
### Changed
* `TypedSchema` and `UntypedSchema` now reject duplicate column names at construction with a validation error, instead of carrying them positionally. A query whose result set has duplicate output column names is still surfaced as a typed `duplicate_columns` execution error, not a crash.
* A scorer paired with the wrong `expected` kind (e.g. `ExecutionAccuracy` without a `GoldQuery`) now returns an inconclusive `ScoreResult` flagged `metadata["scorer_misconfigured"]` instead of raising `TypeError`, so the misconfiguration fails the case visibly rather than aborting the whole run.
### Fixed
* `ResultSetEquivalence` generated SQL that failed on Snowflake and SQLite: the keyless diff used `EXCEPT ALL` (unsupported on both) and keyed comparisons quoted internal aliases that Snowflake case-folds. The scorer now emits portable SQL, verified by conformance tests that run every scorer against every supported engine.
* The Snowflake examples seed into `SNOWFLAKE_DATABASE`/`SNOWFLAKE_SCHEMA` when set, instead of always requiring account-level `CREATE DATABASE` privilege.

## 0.4.0 - 2026-07-04
### Added
* SqliteAdapter: run evals against a local SQLite database via a new `sqlite` platform (in-process, no extra dependencies). Build one with `sqlite_platform(name, path)`; `evaldata doctor --sqlite` checks it.
* ExecutionAccuracy: the standard text-to-SQL execution-accuracy scorer — runs the gold query and the model's SQL and compares the result rows. Its row-order (`when_ordered`/`ignore`), duplicate-handling (`multiset`/`set`), and column-alignment (`by_position`/`by_value`) options reproduce the official Spider and BIRD execution-accuracy comparators, and `evaldata bench` applies the right one per benchmark.
* load_spider() and load_bird(): turn the Spider and BIRD benchmark datasets into runnable eval cases. Run a model over them with run_benchmark(...) or the `evaldata bench <spider|bird>` CLI to get an execution-accuracy number. Use SCHEMA_PROMPT_TEMPLATE to put the database schema in the prompt.
* `evaldata fetch <spider|bird>`: download, verify, and cache a benchmark dataset — pinned to a known SHA-256 with a case-count and SQLite integrity check — so `evaldata bench` can run against it. Downloads are explicit (never during a run).
* Evaluate AI SQL against a dbt project: `load_dbt(...)` reads a built project's `manifest.json` and `catalog.json` into a `DbtContext` — models, sources, resolved column types, and schema context — and builds text-to-SQL eval cases you score against gold queries on your own warehouse, no dbt Cloud account. `platform_from_profile` resolves the warehouse connection from a dbt profile, and `evaldata dbt-bench` runs a model over a cases file. Install the `dbt` extra.
* Evaluate dbt Semantic Layer (MetricFlow) queries: `load_dbt_metrics(...)` reads the semantic manifest's metrics and dimensions and builds metric-query eval cases. `MetricLayerSolver` asks a model to answer each question with a metric query, and `metric_layer_equivalence` scores it with a cascade — resolve-and-compare, run-and-compare, then an LLM judge — that runs the cheapest check first and stops at the first verdict; `MetricLayerJudge` is the judge tier on its own. `evaldata sl-bench` runs it end to end. Install the `dbt-sl` extra (`dbt-metricflow`).
### Changed
* Result schemas are now typed or untyped. `Schema` is a union of `TypedSchema` (columns with types) and `UntypedSchema` (column names only, returned by engines like SQLite whose driver reports no result-column types); type comparison is inconclusive rather than refuting when types are absent. Breaking: `evaldata.types.Schema` is now this union alias, not a class — construct `TypedSchema(...)` directly.

## 0.3.0 - 2026-06-26
### Added
* LlmJudge: an LLM grades the AI's output against criteria you write and returns a pass/fail verdict, a 0-1 score, and a reason. Configure it with criteria, scoring steps, few-shot examples, and a rubric.
* Llm: a single seam for calling language models (litellm under the hood), shared by solvers and the LLM judge, so you can plug in your own model client.
* observed_equivalence(), judged_equivalence(model), and sql_equivalence_judge(model): ready-made query-equivalence scorers that confirm by comparing query structure first, then fall back to running both queries or asking an LLM judge.
* Scorers now return a three-valued verdict — 'pass', 'fail', or 'inconclusive' — alongside an optional 0-1 score, so a scorer can report when it cannot decide rather than being forced to pass or fail.
* basis on every result records how strong the evidence is: 'proven' (sound on all data), 'observed' (seen on the data that was run), or 'judged' (an LLM's judgment).
### Removed
* query_equivalence(): replaced by observed_equivalence(), which runs the same compare-structure-then-run check and can also report 'inconclusive'.

## 0.2.0 - 2026-06-25
### Added
* SemanticEquivalence: checks whether the AI's SQL means the same as a gold query by comparing query structure, without running either query. Reports 'equivalent' or 'unknown'.
* query_equivalence(): checks whether two queries return the same results by running them.
* FirstDecisive: combine scorers and use the first one that reaches a verdict.
### Fixed
* SemanticEquivalence: queries with redundant boolean conditions (e.g. a repeated predicate) no longer cause a slowdown when comparing query structure.
